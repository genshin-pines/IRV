# IRV 双端应用场景与后端设计方案

> 日期：2026-07-08  
> 目标：把小学期任务中较分散的车牌识别、交警手势识别、车主手势控车、Agent 告警，整理成两个清晰应用场景：管理者端与车主端。

---

## 1. 重新定义项目定位

原始任务里包含四类能力：

1. 道路车辆车牌识别
2. 交警手势识别
3. 车主手势控车
4. 日志监控与告警智能体

这些能力直接并列展示会显得项目很散。更合理的产品叙事是：**同一套视觉感知与智能告警底座，服务两个角色端。**

| 端 | 面向对象 | 核心价值 |
| --- | --- | --- |
| 管理者端 | 停车场管理员、交通管理部门、违章处理人员 | 通过道路/停车场摄像头自动识别车牌，辅助停车场管理、车辆追踪、违章记录和异常告警 |
| 车主端 | 普通驾驶员、车载系统用户 | 通过车外/车内摄像头实现辅助驾驶感知、交警手势理解、车主手势控车和驾驶安全预警 |

这样答辩时可以说：

> 本项目不是单纯堆叠算法 demo，而是构建了一个面向智能交通管理与车载交互的双端系统。管理者端关注“道路/停车场监管”，车主端关注“驾驶辅助与车内交互”，二者共用车牌识别、视频流处理、日志监控和 Agent 告警能力。

---

## 2. 双端登录与角色设计

系统设置两种登录身份。

### 2.1 管理者登录

面向：

- 停车场管理员
- 交通管理部门工作人员
- 违章记录审核人员

管理者端本次课程阶段只保留**车牌识别相关功能**，降低实现复杂度，保证主线清晰。

管理者端主要功能：

1. 选择沙盘/道路/停车场摄像头画面。
2. 切换画面后自动启动对应视频流的车牌识别。
3. 实时显示识别到的车牌号、颜色、置信度和出现时间。
4. 记录识别历史，用于停车场车辆记录或违章线索。
5. 当摄像头断连、车牌置信度持续偏低、识别失败过多时，由 Agent 生成告警。

课程限制下的取舍：

- 不做复杂权限体系。
- 不做完整停车缴费系统。
- 不做真实违章判定，只做“违章举报/记录线索”的感知能力展示。
- 管理者端摄像头来源统一使用网站里已有的沙盘 RTSP 视频流。

### 2.2 车主登录

面向：

- 普通驾驶员
- 智能车载系统用户

车主端功能更接近原任务中“车载前置/环视摄像头 + 车内摄像头”的设定。

车主端主要功能：

1. 车外感知：识别前方或周围车辆车牌，便于辅助驾驶、违章举报和行车记录。
2. 交警手势识别：识别停止、直行、左转、右转、减速慢行、靠边停车等交通指挥信号。
3. 车主手势控车：通过车内摄像头识别手掌张开、握拳、画圈、左右滑动、拇指上下、挥手等手势，触发模拟车辆控制。
4. Agent 预警：监听识别结果和系统状态，生成自然语言告警和处理建议。

课程限制下的取舍：

- 真实车外摄像头暂时不用接硬件。
- 车外画面先用网站中的沙盘视频流代替。
- 车内摄像头优先使用本机电脑前置摄像头，用于车主手势控车。
- 自动驾驶控制只做模拟面板，不连接真实车辆。

## 3. 两端功能边界

### 3.1 管理者端功能边界

管理者端只做“外部场景监管”。

| 功能 | 是否做 | 说明 |
| --- | --- | --- |
| 登录为管理者 | 做 | 可以先用简单账号/角色切换 |
| 摄像头列表 | 做 | 展示桥面、停车场入口、道路、隧道等视频流 |
| 切换摄像头 | 做 | 切换后自动识别该路视频流 |
| 实时车牌识别 | 做 | 输出车牌号、颜色、置信度、bbox |
| 识别历史 | 建议做 | 后端可先存 SQLite，前端后续展示 |
| 违章记录 | 轻量做 | 只做“标记为违章线索/异常记录” |
| Agent 告警 | 做 | 摄像头断连、低置信度、连续失败 |
| 交警手势识别 | 不做 | 管理者端不展示，避免场景混乱 |
| 车主手势控车 | 不做 | 管理者端不展示 |

### 3.2 车主端功能边界

车主端做“驾驶辅助 + 车内交互”。

| 功能 | 是否做 | 说明 |
| --- | --- | --- |
| 登录为车主 | 做 | 可以先用角色切换 |
| 前方视频流选择 | 做 | 课程阶段用网站沙盘视频流代替车外摄像头 |
| 前方车牌识别 | 做 | 用于违章举报、辅助驾驶感知 |
| 交警手势识别 | 做 | 映射为交通指挥信号 |
| 本机前置摄像头 | 做 | 代替车内摄像头 |
| 车主手势控车 | 做 | 触发模拟音量、空调、电话、主页等操作 |
| 手势误触发抑制 | 做 | 置信度、持续时间、冷却时间 |
| Agent 告警 | 做 | 低置信度、识别异常、摄像头异常、接口超时 |
| 停车场管理 | 不做 | 这属于管理者端 |

## 4. 后端总体设计

当前 `IRV_main` 后端使用 FastAPI 统一承载所有模块。

```text
IRV_main/
  backend/
    main.py
    routers/
      plate.py
      gesture.py
      cameras.py
      alerts.py
      auth.py          # 后续新增
      records.py       # 后续新增
    services/
      plate_service.py
      gesture_service.py
      camera_service.py
      alert_service.py
      log_service.py
      auth_service.py  # 后续新增
      record_service.py# 后续新增
    models/
      alert_event.py
      recognition_record.py # 后续新增
      user.py               # 后续新增
  alert_agent/
  vendor/
    plate_hyperlpr/
    web_gesture_backend/
  frontend/
```

后端模块职责：

| 模块 | 职责 |
| --- | --- |
| `cameras` | 管理沙盘视频流列表，提供视频流预览与抓拍 |
| `plate` | 对图片、视频、RTSP 流进行车牌识别 |
| `gesture` | 本机摄像头/视频流手势识别，输出手势事件和模拟车辆控制指令 |
| `alerts` | 告警历史、日志查询、模拟异常、WebSocket 推送 |
| `auth` | 双角色登录，区分管理者与车主 |
| `records` | 识别历史、违章线索、驾驶辅助事件记录 |

{

## 6. 管理者端业务流程

### 6.1 摄像头切换与车牌自动识别

管理者进入系统后：

1. 后端返回摄像头列表：

```http
GET /api/cameras
```

2. 管理者选择某个位置，例如停车场入口：

```text
live11 停车场入口
```

3. 前端显示该视频流：

```http
GET /api/cameras/live11/stream
```

4. 同时后端启动对应 RTSP 的车牌识别：

```http
POST /api/plate/recognize-stream
```

请求：

```json
{
  "rtsp_url": "rtsp://10.126.59.120:8554/live/live11",
  "duration_sec": 8,
  "sample_interval": 0.5
}
```

5. 返回识别结果：

```json
{
  "unique_plates": 2,
  "plates": [
    {
      "plate_code": "京A12345",
      "plate_color": "蓝牌",
      "confidence": 0.96,
      "bbox": [100, 220, 280, 270],
      "time_sec": 1.5
    }
  ]
}
```

6. 后端写入日志和识别历史：

```text
plate recognition success camera=live11 count=2
```

7. Agent 监听异常：

- 摄像头断连
- 识别失败
- 置信度持续偏低
- 车牌频繁重复或异常

### 6.2 管理者端最小闭环

```text
管理者登录
  -> 选择摄像头位置
  -> 页面展示该路视频流
  -> 后端自动识别车牌
  -> 保存识别记录/违章线索
  -> Agent 监听异常
  -> 告警推送到管理者端
```

## 7. 车主端业务流程

### 7.1 车外感知

车主端的外置摄像头暂时用沙盘视频流代替。

流程：

```text
车主登录
  -> 选择一个道路视频流作为“前方摄像头”
  -> 系统识别前方车辆车牌
  -> 展示车牌号、颜色、置信度
  -> 可保存为行车记录/违章举报线索
```

对应接口仍然使用：

```http
POST /api/plate/recognize-stream
```

### 7.2 交警手势识别

交警手势识别属于车主端的驾驶辅助功能。

目标输出：

```json
{
  "gesture_type": "stop",
  "gesture_label": "停止信号",
  "confidence": 0.91,
  "suggestion": "前方交警指示停止，请减速停车"
}
```

课程阶段可以先做到：

1. 页面展示“交警手势识别”模块。
2. 用视频流或图片帧输入。
3. 返回识别结果和置信度。
4. 将识别结果写入日志，供 Agent 监听。

### 7.3 车主手势控车

车主手势控车使用本机电脑前置摄像头代替车内摄像头。

支持手势：

| 手势 | 指令 |
| --- | --- |
| 手掌张开 | 启动/唤醒/返回主页 |
| 握拳 | 确认/执行 |
| 单指画圈 | 调节音量 |
| 左右滑动 | 切换功能 |
| 拇指向上/向下 | 接听/挂断电话或调节温度 |
| 挥手 | 返回主页 |

后端接口：

```http
POST /api/gesture/start
```

本机摄像头请求：

```json
{
  "use_webcam": true,
  "camera_index": 0,
  "mirror": true,
  "src_url": null
}
```

视频预览：

```http
GET /api/gesture/video-feed
```

手势事件上报：

```http
POST /api/gesture/events
```

示例：

```json
{
  "source": "driver",
  "gesture_type": "SWIPE_RIGHT",
  "gesture_label": "右滑",
  "confidence": 0.92,
  "stable": true
}
```

返回：

```json
{
  "gesture_type": "SWIPE_RIGHT",
  "command": {
    "target": "turn",
    "action": "right",
    "label": "右转"
  }
}
```

### 7.4 车主端最小闭环

```text
车主登录
  -> 选择道路视频流作为车外摄像头
  -> 识别前方车牌/交警手势
  -> 打开本机前置摄像头作为车内摄像头
  -> 识别车主手势
  -> 触发模拟车辆控制面板
  -> Agent 监听识别异常和系统状态
  -> 给出驾驶辅助或系统告警
```

## 8. Agent 在双端中的作用

Agent 不应该只被描述为“后台日志功能”，而应该作为双端系统的智能运维与安全解释层。

### 8.1 管理者端 Agent

关注系统运行和监管异常：

- 摄像头断连
- 某路视频流长时间无车牌结果
- 车牌识别置信度持续偏低
- 停车场入口车辆识别失败
- 后端接口超时

示例告警：

```text
告警级别：WARNING
来源模块：plate/camera
摘要：停车场入口摄像头在过去 3 分钟内连续出现车牌识别失败，可能由光照不足或视频流卡顿导致。
建议：检查摄像头连接状态，必要时切换备用摄像头。
```

### 8.2 车主端 Agent

关注驾驶辅助和车载交互异常：

- 交警手势置信度低
- 前方车牌识别失败
- 车主手势频繁跳变
- 本机摄像头被占用
- LLM 或识别接口超时

示例告警：

```text
告警级别：INFO
来源模块：gesture
摘要：车主手势识别结果在短时间内多次跳变，可能存在误触发风险。
建议：请保持手势稳定 0.5 秒以上后再执行控制。
```

## 9. 数据记录设计

当前 Agent 已经有 `alert_events` 表。后续建议补三类业务记录。

### 9.1 车牌识别记录

表名建议：`plate_records`

字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `role` | `admin` 或 `driver` |
| `camera_id` | 摄像头编号 |
| `camera_name` | 摄像头名称 |
| `plate_code` | 车牌号 |
| `plate_color` | 车牌颜色 |
| `confidence` | 置信度 |
| `bbox` | 检测框 |
| `source_url` | 视频流或图片来源 |
| `scene` | `parking`、`road`、`driver_front` |
| `created_at` | 创建时间 |

### 9.2 手势事件记录

表名建议：`gesture_events`

字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `role` | 通常为 `driver` |
| `gesture_type` | 手势类型 |
| `gesture_label` | 中文标签 |
| `confidence` | 置信度 |
| `stable` | 是否稳定 |
| `command_target` | 控制目标 |
| `command_action` | 控制动作 |
| `created_at` | 创建时间 |

### 9.3 违章/异常线索记录

表名建议：`incident_records`

字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `role` | 由谁创建 |
| `source` | `admin` 或 `driver` |
| `plate_code` | 相关车牌 |
| `incident_type` | 违章线索、停车异常、摄像头异常等 |
| `description` | 描述 |
| `status` | 待处理、已确认、已忽略 |
| `created_at` | 创建时间 |

## 10. 当前课程阶段实施优先级

### 第一阶段：双端叙事打通

必须完成：

1. 登录页或角色选择页：管理者 / 车主。
2. 管理者端只展示摄像头切换 + 车牌识别 + 告警。
3. 车主端展示车外感知 + 交警手势 + 车主手势控车 + 告警。
4. 前端视觉上明确两个场景，不要所有功能挤在一个页面。

### 第二阶段：后端记录补齐

建议完成：

1. 车牌识别记录入库。
2. 手势事件入库。
3. 告警与识别记录关联。
4. 管理者端可查看历史车牌记录。
5. 车主端可查看驾驶辅助历史事件。

### 第三阶段：答辩增强

可选完成：

1. 管理者端“违章线索标记”。
2. 车主端“模拟车辆控制面板”。
3. Agent 的自然语言解释更贴近双端场景。
4. 演示脚本设计成两条故事线。

## 11. 推荐答辩故事线

### 11.1 管理者故事线

> 停车场管理员登录系统，选择“停车场入口”摄像头。系统自动拉取该路视频流并识别进出车辆车牌。当摄像头出现断连或识别置信度持续偏低时，Agent 自动生成告警，提示管理员检查摄像头或光照环境。识别记录可以作为停车管理和异常车辆追踪依据。

### 11.2 车主故事线

> 普通驾驶员登录车主端，系统使用道路视频流模拟车外摄像头，识别前方车辆车牌并辅助记录异常车辆。同时，系统识别交警指挥手势，帮助驾驶员理解交通指令。驾驶员还可以通过本机前置摄像头模拟车内摄像头，用手势完成车窗、音量、电话等非接触式控制。Agent 持续监听识别置信度、手势跳变和摄像头状态，及时给出安全提示。

## 12. 最终设计结论

本项目应被包装为：

> 面向交通管理者与普通车主的双端智能车载视觉感知系统。

管理者端强调：

- 摄像头监管
- 车牌识别
- 停车场/违章线索
- Agent 运维告警

车主端强调：

- 驾驶辅助
- 前方车牌记录
- 交警手势理解
